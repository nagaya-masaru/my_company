## 3. 三層モデル — A/B/C の連動

```
            ┌─────────────────────────────────────────┐
            │ CEO + Claude 20エージェント (経営チーム)        │
            └─────────────┬───────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
   ┌─────────┐      ┌─────────┐       ┌─────────────┐
   │ A 層    │      │ B 層    │       │ C 層 (Roll Up)│
   │ SaaS    │      │ 受託    │       │ 事業再生 +     │
   │ 月額    │      │ 一括    │       │ エクイティ参画 │
   │ 中堅    │      │ 知人    │       │ 地方中小      │
   └────┬────┘      └────┬────┘       └──────┬───────┘
        │                │                   │
        └────────┬───────┴───────────────────┘
                 ▼
         共通: 20エージェント基盤 / RAG / Knowledge Graph
                 │
                 ▼
   ┌────────────────────────────────────────┐
   │   Roll Up Co. (持株会社 or PEファンド)         │
   │   → IPO / Strategic Sale / Secondary    │
   └────────────────────────────────────────┘
```

### 3-1. なぜ三層が成立するか
- **A層** が立ち上がる → **当社のブランド・信用** が形成される
- **B層** で **個別中小企業の実装ノウハウ** がテンプレ化される
- **C層** で **株式を取得し、A/B層のノウハウで価値向上** させる
- 全層が **共通の20エージェント基盤 + 顧客文脈RAG** を共有する
- 単独事業の3層連動シナジーが、競合 (純粋なPE・純粋なSaaS) には作れない

### 3-2. C層 → Roll Up の意思決定フロー

1. **無料診断 / クイック改善** で接触
2. **月額伴走** で社内入り (3-6ヶ月、CFO/COO代行)
3. 数字が見える状態に → **改善余地特定**
4. **エクイティ型へ移行**: 株式 5-30% を取得 (キャッシュ + 改善実績で評価)
5. **当社が経営に深く関与**: AI実装+販路+人材
6. 数年後、**他Roll Up子会社と統合 or 単独売却**

### 5-3. ICP (Roll Up 対象企業)

**S級ターゲット**:
- 業種: 製造業 / 自動車整備 / 不動産管理
- 売上: 1-10億
- 営業利益: 500万-3,000万 (利益薄いが赤字ではない)
- 経営者年齢: 55歳以上 (事業承継不安)
- 借入: 売上の50%以下 (健全範囲)
- 立地: 地方都市 (東京・大阪以外も歓迎)
- 売却意思: 後継者不在 or 早期Exit希望

**避けるべき企業** (CEO docs §7 と整合):
- 経営者に危機感がなく外部介入を拒む
- すでに基幹システムが重く、AI実装余地がない
- 売上基盤がなく市場自体が衰退
- 法務・税務リスクが高い (簿外債務、訴訟係争中)

### 5-4. エクイティ型契約スキーム ([[legal-komikado]] [[legal-kujou]] 設計)

#### Phase 1 (Year 1-2): マイノリティ参画
- 株式 **5-15%** を取得
- 評価方法: 純資産+営業利益×3-5倍の混合
- 対価: **キャッシュ 30% + 改善約束 (Earn-in) 70%**
  - 改善約束 = 営業利益を1.5倍にしたら追加株式取得
- 役員派遣: 当社から非常勤取締役1名 (CEO自身 or 設楽)

#### Phase 2 (Year 3-5): 過半数取得 → 完全買収
- 業績が改善した子会社の株式を **追加取得して51-100%**
- Earn-in での無償取得分 + 追加買取
- 経営権獲得後、PMI で当社プラットフォームに統合

#### Phase 3 (Year 5+): Roll Up 統合 → Exit
- 同業セグメント内で複数子会社を統合
- 統合後の合計企業価値で売却 (個別売却より3-5倍高い)

## 6. 技術アーキテクチャ (Roll Up を支えるシステム)

### 6-1. Torvalds の宣言
> 「Roll Up は事業の話だが、それを動かすのは技術だ。
> 30社を統合するのに30の業務システムを別々に運用するのは正気の沙汰じゃない。
> 共通基盤を最初から作る。これが Schwarzman の戦略の成否を決める。」

### 6-2. Ritchie の宣言
> 「3年で動くシステムではなく、10年で30社を支えるアーキテクチャを設計します。
> Unixの哲学: Make each program do one thing well. 各機能を単機能で組み合わせ可能に。
> データモデルを綺麗にすれば、コードは自ずから綺麗になります。」

### 6-3. アーキテクチャ全体図

### 6-4. 技術スタック (Torvalds + Ritchie + Altman 合意)

| レイヤー | 採用技術 | 採用理由 |
|---------|---------|---------|
| Frontend | Next.js 15 + Server Components | SEO + 動的UI、Vercelで即デプロイ |
| Backend | Python (FastAPI) + Rust (高性能部分) | LLM周辺はPython、データ処理はRust |
| LLM API | Anthropic Claude (Primary) + OpenAI (Fallback) | プロバイダ依存を回避 |
| Vector DB | Pinecone (Phase 1) → Self-host Weaviate (Phase 3) | 立ち上げは速度、規模が出たらコスト最適化 |
| Knowledge Graph | Neo4j Aura | 顧客企業の関係性可視化 |
| Auth/Multi-Tenant | Supabase Auth + RLS | 子会社単位の権限分離が容易 |
| Database | PostgreSQL (Supabase) + pgvector | Pinecone卒業時に移行容易 |
| Event Bus | NATS JetStream | 軽量、子会社間のイベント連携 |
| Billing | Stripe + Internal Allocation | 子会社間の社内振替対応 |
| Audit Log | Immutable Append-only Store (S3) | 法務・PMI時のエビデンス |
| Infra | Vercel + GCP (Tokyo) + Cloudflare | 国内法準拠 + 海外展開可 |
| 監視 | Datadog + Sentry | 30社規模で人手では追えない |

### 6-5. AI Roll Up 固有の技術要件

#### 6-5-1. Multi-Tenant 設計 (Ritchie主導)
- 各子会社 = 独立テナント (Tenant ID で完全分離)
- データ漏洩を構造的に防ぐ (Supabase RLS + Anthropic Workspaces)
- 共通基盤の更新が全テナントに同時反映できる仕組み
- テナント間の **匿名化されたベンチマーク** (例: 「製造業ロールアップ内で粗利率Top 3」)

#### 6-5-2. 業種別エージェントテンプレート (Altman主導)
- 製造業エージェント / 建設業エージェント / 自動車整備エージェント...
- 各業種の docs §3 (CEO作成) のペインポイント・施策・KPI をエージェントに焼き込む
- 「業種別Claude」として、買収後即運用可能な状態にする

#### 6-5-3. CFO/COO 代行エージェント
- 子会社のCFO/COOを当社のClaude エージェントが代行
- リアルタイム財務ダッシュボード
- 月次経営会議の自動議事録 + アクション項目抽出
- 法務 (NDA・契約期限・許認可) の自動アラート

#### 6-5-4. PMI 専用エージェント
- M&A 直後の100日プラン自動生成
- 共通基盤への移行ウィザード (会計・人事・販売管理データの移行)
- 旧経営陣と新経営方針のすり合わせドラフト
- 従業員向けコミュニケーション素材自動生成

#### 6-5-5. Roll Up Knowledge Graph
- 全子会社の顧客・取引先・仕入先・人材を Neo4j で繋ぐ
- クロスセル機会の自動発見 (子会社A の顧客が子会社B の見込み客)
- 競合・代替企業のマッピング
- 業界全体の「次の買収候補」を Schwarzman に提示

### 6-6. 技術的に対応可能なこと (洗い出し一覧)

CEO質問「技術的に対応するのであればどのようなことが可能か」への網羅的回答:

#### A. 業務代行 (即時実装可能 — 2026年内)
1. **議事録自動生成** (会議録音 → 要約 → アクション抽出)
2. **メール一次対応** (受信メールの分類・返信案生成・送信前承認)
3. **契約書チェック** (条項リスク抽出、修正案提示)
4. **請求書照合** (発注 vs 納品 vs 請求の整合チェック)
5. **日報集計** (現場日報フォーム入力 → 数字を自動集計)
6. **シフト作成** (希望と稼働実績から最適シフト案)
7. **見積書テンプレート + 過去案件参照** (RAG)
8. **電話対応 (受付・FAQ)** (Twilio + Claude Voice)

#### B. 経営分析 (2026-2027 実装)
9. **製品別・取引先別粗利分析** (docs §3-1 製造業ペイン15) を全自動化
10. **資金繰り表自動生成** (銀行明細+売掛買掛から将来CFを予測)
11. **販路別収益分析** (POS/EC/卸の売上を統合)
12. **競合動向自動収集** (Web/SNS スクレイピング + 要約)
13. **取締役会用 月次経営レポート自動生成**
14. **KPI ダッシュボード (子会社別+全体)**

#### C. M&A支援 (2027実装)
15. **DD自動化** (Financial DD/Commercial DD の80%を自動)
16. **企業評価額自動算定** (DCF/EBITDA倍率/類似企業比較を統合)
17. **ターゲット企業の事業価値スコアリング** (Claude による定性評価+定量データ統合)
18. **PMI 100日プラン自動生成**

#### D. 高度ロールアップ運用 (2028以降)
19. **シナジー機会の自動発見** (Knowledge Graphベース)
20. **子会社間最適価格決定** (内部振替価格の自動算出)

### 6-7. 「技術的に対応できないこと」も洗い出す (Hinton/Ritchie 共同)

CEOへの誠実な開示として、以下は **技術的に困難** または **慎重な扱いが必要**:

| 機能 | 困難の理由 | 対処 |
|------|-----------|------|
| 完全自動の経営判断 | 責任所在が法的に不明確 | 必ず人間 (CEO/取締役会) が最終決裁 |
| 個人情報を含む顧客対応の完全自動化 | 個人情報保護法 | 人間承認フローを残す |
| 製造現場のIoT統合 (旧設備) | レガシー機器の通信規格対応コスト | 段階導入、または現場の手動入力フォームで代替 |
| 100% 多言語対応 (Phase 1) | コストとQA時間 | 日本語+英語に絞る、他言語は Year 3 以降 |
| ハードな金融商品の販売 | 金商法・士業独占 | 該当領域は商品ラインから除外 |

## 8. 財務モデル詳細 ([[finance-buffett]] [[investment-marks]] 共同)

### 8-1. Buffett 絶対条件 (Decision 0003 から継承)
1. 本業 (A層) の単月黒字化 Month 3 まで Roll Up 資金は使わない
2. 手元キャッシュ 1,200万 (Year 1) / 5,000万 (Year 2) を割り込む買収は不可

## 9. 規制・法務リスク (古美門・九条 共同レビュー)

### 9-1. 主要規制リスク

| リスク | 影響 | 対処 |
|--------|------|------|
| 金商法 (経営助言の対価が金融商品助言になる) | 中 | エクイティ参画は「事業投資」として整理、金商法に抵触しない契約構造 |
| 士業独占 (税理士法・行政書士法) | 中 | 提案では士業領域に踏み込まない、士業と協業する形にとどめる |
| 独占禁止法 (同業ロールアップによる市場集中) | 低 (規模が小さいうち) | 業種ごと売上シェア20%超を超える前に公取と非公式対話 |
| 個人情報保護法 (子会社統合時) | 中 | データ統合は同意取得 + 匿名化 |
| 労働法 (PMI時のリストラ) | 中 | リストラは原則しない。配置転換+早期退職優遇で対応 |
| 大量保有報告 / 開示規制 (上場準備時) | 高 | 上場2年前から監査法人・主幹事証券と連携 |

### 9-2. 古美門スタンス
> 「M&A は私の得意領域だ。
> AI系子会社買収のLegal DD は通常の3倍時間がかかる:
> - データ取得経路の正当性 (Webスクレイピング訴訟リスク)
> - モデル学習に使用したデータの著作権
> - 既存顧客との契約のNo Assignment条項
> - 役員・主要エンジニアのリテンション (Earn-out設計)
> 軽く見ると、買収後3年で爆発する」

### 9-3. 九条スタンス
> 「私が懸念するのは、買収先創業者の『心』だ。
> 売却するのは『万策尽きた』ケースが多い。
> 紳士的に振る舞う技術が必要だ。私はその技術を持っている。
> 古美門の華やかな宴会では片付かない案件の方が、後で安く買える」
